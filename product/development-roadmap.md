## 1. 项目目标与当前阶段

本项目目标是建设面向湘潭大学新生和在校学生的学习文档知识库，第一版以网页站点交付，并把 RAG 问答作为核心能力纳入 MVP。

当前阶段不是直接堆功能，而是先把“资料结构、页面范围、开发节奏”定清楚，再开始工程初始化。朋友们，这一步非常关键：先把地基打稳，后面开发才能快、准、稳。

## 3. MVP 范围锁定

### 3.1 第一版必须完成

- Next.js 网页站点基础工程
- 首页
  - 新生入学路线
  - AI 问答主入口
  - 联系方式 / 加群咨询入口
  - QQ 新生群、湘大表白墙二次强调卡片
- 入学指南页面
  - 报到流程
  - 宿舍生活
  - 军训指南
  - 学习入门
- 资料库页面
  - 沿用本地 `docs/` 分类导航
  - 只展示已审核状态资料
- Markdown 内容维护体系
- 本地 RAG 验证闭环
  - 基于已审核 Markdown 资料检索
  - 回答必须引用来源
  - 资料不足时不编造，提示加群 / 联系补充

### 3.2 第一版暂不做

- 复杂后台管理系统
- 多人投稿审核流
- 独立“联系我们 / 加群咨询”页面
- 生产级云端 RAG 部署
- 未审核资料直接进入正式资料库或索引

## 4. 开发阶段拆分

### Phase 0：资料接收与内容结构确认

目标：把用户提供资料转化为可开发、可维护的内容结构。

任务：

1. 确认腾讯文档导出文件完整性。
2. 基于提取文本梳理内容栏目映射。
3. 将资料分为“新生入学”“校园生活”“课程学习”“资源工具”“组织社团”等内容域。
4. 设计 Markdown 文档 frontmatter 字段：标题、分类、状态、更新时间、来源、是否可索引。
5. 等待用户确认哪些内容可以进入正式资料库。

交付物：

- `docs/99-待整理资料/湘潭大学2026新生攻略.docx`
- `docs/99-待整理资料/湘潭大学2026新生攻略-提取文本.md`
- 内容栏目映射规则

验收标准：

- 资料不直接进入 RAG 索引。
- 待整理资料有来源说明和审核状态提示。
- 能明确哪些内容支撑首页、入学指南和资料库。

### Phase 1：Next.js 工程初始化与基础骨架

目标：搭建能运行、能浏览、能继续扩展的站点工程。

任务：

1. 初始化 Next.js 项目。
2. 配置 TypeScript、基础样式方案与代码规范。
3. 建立基础路由：`/`、`/guide`、`/library`。
4. 创建全站布局组件：顶部导航、联系方式弹窗、页脚。
5. 建立内容读取工具，后续从 Markdown 读取资料。

交付物：

- 可运行的 Next.js 工程
- 首页 / 入学指南 / 资料库基础页面
- 顶部导航与联系方式弹窗雏形

验收标准：

- 本地开发服务可启动。
- 三个核心页面可访问。
- 顶部导航入口符合 Round 10 规则。

### Phase 2：首页与核心页面实现

目标：完成可演示网页 MVP 的主体验。

任务：

1. 首页实现新生入学路线模块卡片。
2. 首页实现 AI 问答主入口的前端占位或本地交互入口。
3. 首页实现 QQ 新生群、湘大表白墙二次强调卡片。
4. 入学指南实现统一模板。
5. 资料库实现分类导航，先按目录展示资料。

交付物：

- 首页完整首版
- 入学指南统一模板页
- 资料库分类页

验收标准：

- 用户能从首页快速进入报到流程、宿舍生活、军训指南、学习入门。
- 联系方式入口在顶部导航和首页均可见。
- 页面移动端可基本浏览。

### Phase 3：Markdown 内容体系与审核状态展示

目标：让资料可以被站点稳定读取，并为 RAG 做准备。

任务：

1. 建立正式资料 Markdown 模板。
2. 为每篇资料增加审核状态、来源说明、更新时间。
3. 实现资料页面“已审核”状态展示。
4. 建立待整理资料与正式资料的迁移规则。
5. 将用户确认后的内容逐步拆分入正式目录。

交付物：

- Markdown 内容模板
- 资料读取与展示逻辑
- 已审核标识

验收标准：

- 未审核资料不会出现在正式资料库页面。
- 已审核资料可被页面展示。
- 文档结构支持后续句子级引用。

### Phase 4：RAG 本地验证闭环

目标：在本地完成“资料检索 → 回答生成 → 来源引用 → 失败兜底”的最小闭环。

任务：

1. 评估并接入 `raganything` 技术路线。
2. 设计本地索引构建脚本。
3. 只索引已审核资料。
4. 实现问题输入、检索、回答、引用来源展示。
5. 实现资料不足时的兜底提示。

交付物：

- 本地 RAG 验证脚本或服务
- 首页 AI 问答入口对接本地 RAG
- 引用来源展示规则

验收标准：

- 能基于已审核资料回答典型问题。
- 答案包含来源引用。
- 无依据问题不会编造答案。

### Phase 5：UI 打磨、移动端适配与验收

目标：把 MVP 从“能用”提升到“愿意用”。

任务：

1. 统一年轻活泼视觉风格。
2. 优化首页卡片、导航、联系方式弹窗。
3. 优化移动端阅读体验。
4. 整理验收清单与演示流程。
5. 记录已知问题和下一版计划。

交付物：

- 可演示 MVP
- 验收清单
- 下一版计划

验收标准：

- 首页、入学指南、资料库、AI 问答入口能完整走通。
- 移动端无明显排版阻塞。
- 用户可以基于演示版本继续补充资料。

## 5. 优先级排序

| 优先级 | 任务 | 原因 |
| --- | --- | --- |
| P0 | 资料接收与审核边界 | 防止未经确认资料进入正式库或 RAG |
| P0 | Next.js 工程初始化 | 所有页面与交互的基础 |
| P0 | 首页 + 入学指南 + 资料库骨架 | MVP 核心浏览路径 |
| P0 | Markdown 内容模板 | 决定后续维护和 RAG 可追溯能力 |
| P1 | 联系方式弹窗与首页二次强调 | 已确认的全站转化路径 |
| P1 | RAG 本地验证 | 第一版核心亮点，但依赖资料结构 |
| P2 | UI 打磨与移动端优化 | 提升可用性和传播效果 |
| P2 | 云端部署准备 | 本地闭环稳定后再推进 |

## 7. 风险与依赖

1. 腾讯文档导出内容可能不完整：当前 DOCX 初步提取文本较少，后续可能需要用户补充更完整导出或逐页展开后再提取。
2. 资料真实性依赖用户审核：即使资料来自用户提供链接，也应继续遵守“审核后入库”。
3. RAG 技术依赖资料结构：如果 Markdown 结构不稳定，后续引用溯源会变复杂。
4. UI 风格尚未细化：年轻活泼方向已确认，但色彩、组件、动效仍需后续对齐。
5. 联系方式内容需要最终素材：微信二维码、QQ 群链接、湘大表白墙入口需要用户提供最终可展示版本。

## 9. 当前开发进程汇报规则

后续每次阶段性回复都必须包含：

1. 当前进程：说明处于哪个阶段，以及该阶段完成度。
2. 本次完成：列出本轮实际完成的文件和能力。
3. 验证状态：说明是否运行 `npm run build`、`npm run rag:index` 或其他验证。
4. 下一步建议：只给 1-2 个最优先方向，避免分散。
5. 轻量约束：说明本轮是否避免了不必要的内容和代码膨胀。

## 10. 轻量开发约束

1. 单次迭代优先小步快跑，避免一次性新增大量资料或复杂组件。
2. 先复用现有 `app/`、`components/`、`lib/` 结构，不为相似功能重复造组件。
3. 内容入库优先保证准确、可引用、可回滚，而不是追求数量。
4. RAG 优化优先使用本地索引和评分策略，真实模型接入单独规划。
5. 每一轮都要保持开发日志和路线图同步更新。

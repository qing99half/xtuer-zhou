## 1. 产品背景

用户希望建设一个类似飞书知识库的湘潭大学学习文档项目，用于后续承接、整理和展示补充资料。

当前不进行爬虫和资料下载，仅保留本地目录结构，并通过轮询方式逐步确认 PRD。

## 2. 已确认需求

### 2.1 项目对象

- 学校对象：湘潭大学
- 项目类型：学习文档知识库
- 参考形态：类似飞书知识库
- 当前资料状态：后续由用户补充资料

### 2.2 当前目录

### 2.3 Round 1：产品定位

- 第一目标用户：新生优先
- MVP 阶段最重要场景：入学指南
- 最终形态：AI 问答库

### 2.4 产品定位结论

本项目优先服务湘潭大学新生，先解决入学阶段的信息差问题，包括报到、宿舍、军训、校园生活、常用系统、办事流程等高频问题。MVP 阶段以“入学指南”为核心内容场景，先沉淀结构化资料；长期目标是在资料库基础上升级为 AI 问答知识库，让用户可以直接提问并获得可信答案。

### 2.5 Round 2：内容范围

- MVP 第一批内容模块：报到流程、宿舍生活、军训指南、学习入门
- 内容呈现形式：长文指南 + 问答卡片两者结合
- AI 答案来源要求：必须引用资料原文

### 2.6 内容范围结论

MVP 阶段的新生入学指南围绕 4 个一级模块展开：

1. 报到流程：解决“什么时候来、带什么、怎么到、到哪办”的入校第一天问题。
2. 宿舍生活：解决住宿、生活用品、校园卡、水电空调、快递等高频生活问题。
3. 军训指南：解决军训安排、物资准备、注意事项、请假与经验问题。
4. 学习入门：解决教务系统、选课、培养方案、课程表、学分、常用学习平台等起步问题。

内容组织采用“两层结构”：

- 长文指南：用于完整讲清楚流程、背景和注意事项。
- 问答卡片：用于承接高频问题，并为后续 AI 问答库提供结构化语料。

AI 问答库必须引用资料原文或明确资料来源，避免无依据回答。政策、时间、地点、流程、材料清单等高风险信息尤其必须可追溯。

### 2.7 Round 3：MVP 交付形态

- MVP 第一版交付形态：网页站点
- AI 问答能力：现在规划
- 内容更新权限：个人维护

### 2.8 MVP 交付形态结论

MVP 第一版不只停留在本地 Markdown 目录，而是以可浏览的网页站点作为主要交付形态。网页站点需要优先满足新生快速查阅入学指南的需求，内容仍由本地结构化文档驱动，后续可逐步演进为正式知识库产品。

AI 问答能力从当前阶段开始纳入规划，但不要求第一版完整实现。当前需要先设计好资料结构、引用规范、问答卡片格式和内容元数据，确保后续接入 AI 问答时可以做到“有来源、可追溯、可维护”。

内容维护权限在 MVP 阶段采用个人维护模式，先保证内容质量、结构统一和可信来源。后续资料量增加后，再评估小组协作或投稿审核机制。

### 2.9 Round 4：网页站点范围

- 第一版页面范围：首页、入学指南、资料库
- 全站显眼联系方式：微信二维码、QQ 新生群、湘大表白墙
- 首页重点：新生入学路线
- 后续亮点：补充 RAG 检索能力
- 视觉风格：年轻活泼，开发过程中需要持续与用户详细对齐

### 2.10 网页站点范围结论

网页站点第一版必须包含 3 类核心页面：

1. 首页：突出新生入学路线，帮助新生按阶段理解“录取后—报到前—到校当天—入校后”的关键事项。
2. 入学指南：承载报到流程、宿舍生活、军训指南、学习入门 4 个 MVP 内容模块。
3. 资料库：用于按分类浏览资料，为后续扩展学习文档和 RAG 检索做内容底座。

每个页面都必须在标题栏或其他显眼位置展示项目联系方式与社群入口，包括：

- 微信二维码
- QQ 新生群
- 湘大表白墙

首页当前以“新生入学路线”为主，但需要在产品规划中预留 RAG 检索能力，并将其作为后续项目亮点。视觉风格采用年轻活泼路线，但具体 UI 方案、色彩、排版、组件风格需要在后续开发过程中与用户逐轮对齐。

### 2.11 Round 5：站点导航、首页结构与 RAG 要求

- 联系方式展示方式：顶部导航
- 首页“新生入学路线”呈现方式：模块卡片
- RAG 检索要求：第一版必须完成，不仅预留入口
- RAG 技术支撑：后续开发中引用 `raganything` 项目作为技术支撑
- 项目节奏：允许为 RAG 能力投入一定开发时间，不强行压缩为纯静态 MVP

### 2.12 导航与 RAG 结论

全站联系方式统一放在顶部导航栏的显眼位置，确保用户在首页、入学指南、资料库等页面都能快速找到微信二维码、QQ 新生群和湘大表白墙入口。

首页“新生入学路线”采用模块卡片形式呈现，优先突出报到流程、宿舍生活、军训指南、学习入门 4 个核心模块。模块卡片需要具备明确标题、简短说明、关键问题入口和跳转按钮，降低新生理解成本。

RAG 检索能力不再作为远期预留功能，而是纳入第一版必须完成的核心能力。后续开发阶段将评估并引用 `raganything` 项目作为技术支撑，围绕本地资料解析、索引构建、检索增强生成、引用溯源等能力进行落地。第一版 RAG 必须满足：

1. 能基于已整理资料进行检索。
2. 能生成面向新生问题的回答。
3. 回答必须引用资料来源。
4. 对无资料支撑的问题，应提示“资料库暂未覆盖”，不能编造答案。

### 2.13 Round 6：RAG 第一版能力边界

- RAG 第一版资料来源范围：全部资料库
- RAG 问题入口：首页主入口
- RAG 回答失败或资料不足时的引导：引导添加/使用联系方式进行咨询或补充资料

### 2.14 RAG 第一版能力边界结论

RAG 第一版不只覆盖入学指南，而是覆盖站点“资料库”中的全部已整理资料。所有进入资料库的文档都需要按照可检索、可引用、可追溯的标准管理，确保 RAG 能够基于完整资料范围进行回答。

首页需要设置醒目的 AI 问答主入口，把 RAG 检索作为项目核心亮点展示。用户进入站点后，应能直接在首页输入问题，例如“报到要带什么”“宿舍床多大”“军训要准备什么”“选课系统在哪”。

当资料不足、检索不到可靠依据或问题超出知识库覆盖范围时，系统不能编造答案，应明确提示资料库暂未覆盖，并引导用户通过顶部导航中的微信二维码、QQ 新生群、湘大表白墙等联系方式继续咨询或补充资料。

### 2.15 Round 7：技术路线与开发边界

- 网页站点技术栈：Next.js
- RAG 服务第一版部署方式：先本地后云端
- 第一版资料维护方式：文件维护

### 2.16 技术路线结论

网页站点采用 Next.js 作为第一版技术栈，便于同时承载内容页面、首页交互、后续 API 接口和 RAG 能力集成。第一版优先完成可访问网页站点与 RAG 问答闭环，而不是先做复杂后台。

RAG 服务采用“先本地后云端”的路线：

1. 本地阶段：优先验证资料整理、索引构建、检索问答、引用溯源和 raganything 接入可行性。
2. 云端阶段：在本地流程稳定后，再考虑部署到服务器或云平台，支撑真实用户访问。

资料维护在第一版采用文件维护方式，优先使用 Markdown、JSON 或结构化配置文件管理内容，符合个人维护、高质量整理、快速迭代的 MVP 目标。后台管理页面不进入第一版刚需，但文档结构应尽量保持规范，方便后续迁移到后台或 CMS。

### 2.17 Round 8：资料文件结构与引用规范

- 资料正文第一版文件结构：Markdown
- 引用来源粒度：句子级
- 入库审核机制：所有资料由用户先行审核后再入库

### 2.18 资料结构与引用规范结论

第一版资料正文采用 Markdown 文件维护，降低资料录入与后续整理成本。Markdown 文件需要保持标题层级清晰、段落短小、语义明确，方便网页展示、RAG 切分、句子级引用和人工校对。

RAG 引用来源需要精确到句子级。系统在回答用户问题时，应尽量给出具体原文句子或可定位到原文句子的引用信息，而不是只引用整篇文档。这样可以提升答案可信度，并便于用户核对来源。

所有资料入库前都需要由用户先行审核。未审核资料不得进入正式资料库与 RAG 索引，可以先放入 `docs/99-待整理资料/`。审核通过后，再归档到对应栏目并进入可检索资料范围。

### 2.19 Round 9：页面信息架构与内容状态

- 资料页面内容状态：展示“已审核”
- 入学指南 4 个模块模板：统一模板
- 资料库分类导航：沿用当前目录

### 2.20 页面信息架构结论

资料页面第一版只展示“已审核”状态，避免状态体系过重。所有进入正式资料库与 RAG 索引的资料都必须经过用户审核，并在页面中明确展示“已审核”，提升用户对内容可信度的感知。

入学指南中的报到流程、宿舍生活、军训指南、学习入门 4 个模块采用统一模板，保证页面体验一致、内容结构稳定、后续维护简单。统一模板建议包含：

1. 模块简介
2. 核心问题卡片
3. 长文指南正文
4. 关键材料 / 注意事项
5. 常见问答
6. 引用来源与更新时间

资料库分类第一版沿用当前本地目录结构，确保文件维护、网页导航和 RAG 入库逻辑一致，减少额外映射成本。

### 2.21 Round 10：联系方式内容与展示规则

- 顶部导航联系方式展示方式：文字入口 + 二维码弹窗
- QQ 新生群和湘大表白墙：首页卡片中二次强调
- 联系方式单独页面：第一版暂不设置

### 2.22 联系方式展示规则结论

全站顶部导航保持简洁，只展示“联系方式 / 加群咨询”等文字入口。用户点击后，以弹窗形式集中展示微信二维码、QQ 新生群、湘大表白墙等联系信息，避免在导航栏直接展开大量二维码或说明文字。

首页需要通过卡片或模块二次强调 QQ 新生群和湘大表白墙，帮助新生在浏览入学路线时快速找到咨询入口。二次强调不改变首页以“新生入学路线”为主的结构，而是作为辅助咨询入口嵌入页面。

第一版暂不单独设置“联系我们 / 加群咨询”页面，避免 MVP 页面结构过重。后续如果联系方式、投稿、资料补充入口变复杂，再升级为独立页面。

## 3. 开发约束与资料来源规范

### 3.1 禁止未经确认的网络信息入库

开发过程中禁止将未经用户确认的网络信息来源写入项目存储、正式资料库或 RAG 索引。

具体要求：

1. 不得直接把搜索引擎、网页、论坛、公众号、社群截图等网络信息写入正式资料库。
2. 不得使用未经用户确认的网络内容作为 RAG 回答依据。
3. 外部资料必须先放入待整理区，经过用户审核确认后，才可进入正式资料库。
4. AI 生成内容不得直接作为事实资料入库，必须有用户确认或可信原文来源支撑。
5. 若资料来源不明确，应标记为待确认，不进入 RAG 索引。

### 3.2 轻量开发与汇报约束

后续开发必须控制代码和内容体量，优先做能验证主流程的最小闭环，避免过早堆叠大量页面、组件、资料和复杂依赖。

具体要求：

1. 每次新增内容前，先判断是否支撑当前 MVP 主流程；不支撑的内容先记录到待办，不立即入库。
2. 每次新增代码前，优先复用现有组件、样式变量和数据结构，避免重复创建相似组件。
3. 单次资料入库优先小批量、可验证、可回滚；不一次性拆分过多长文。
4. RAG 优化优先提升检索质量和引用可信度，不急于引入复杂模型服务。
5. 每次阶段性回复都必须说明当前开发进程、已完成内容、下一步建议和验证状态。

## 4. 待轮询确认

### Round 11：RAG 第一版交互与回答规则

待确认问题：

1. 首页 AI 问答入口采用单输入框、搜索框 + 示例问题，还是完整问答面板？
